SYSTEM CONTEXT

El problema prioritario

Bypass técnico de QueueIT mediante cookie, URLs directas y acceso sin token válido. Impide que socios legítimos accedan a entradas.

Evidencia en Data

  • 60-70% de ventas concentradas en primeros 5 min.
  • Pico artificial, no refleja salida real de QueueIT.
  • Tiempo promedio de compra <2 min (indica bots).

Para quién resolvemos

  • Prioridad 1: Socio activo sin filtro de asistencia que compite contra bots.
  • Prioridad 2: Socio con filtro que necesita mejor comunicación.
KERNEL

Objetivo 2026

Cerrar los vectores de bypass de QueueIT y lograr que las ventas reflejen el orden real de acceso de socios legítimos.

Métrica Norte: Distribución temporal en primeros 20 minutos

Baseline actual~60-70% en 5 min
Target Q2<30% en 5 min
Target Q4<25% en 5 min

Detección Bypass

Cookies compartidas <2%
Bypass stepper bloqueado 100%
Tokens inválidos bloqueados 100%

Performance Tech

Sesiones sin error >95%
Tiempo checkout 3-5min
Errores 5XX gateway <2%

Experiencia Socio

Tickets soporte errores -30%
Visibilidad transacciones 100%

KERNEL · APUESTAS

3 hipótesis core

¿Qué creemos que va a mover la aguja?

1. Cookie Binding

Si asociamos cookie ↔ socio_id, eliminaremos el 80% del fraude por cookie compartida.

Acción: Validar binding en cada request

Timeline: Q1 — Ya implementado

2. Stepper Obligatorio

Si forzamos pasos secuenciales, eliminaremos acceso directo a urls de checkout.

Acción: State machine + invalidación token

Timeline: Q1 — Implementar y testear

3. Validación QueueIT

Si validamos token en TODOS los endpoints críticos, cerramos bypass de fila.

Acción: Middleware global de validación

Timeline: Q1 — Implementar

KERNEL · FILTRO

Matriz Go / No-Go

Toda propuesta pasa por este filtro antes de entrar al pipeline.

CriterioGOCondicionalNO GO
Impacto anti-fraudeReduce bypass / reventa directoImpacto indirecto a validarSin impacto claro
Viabilidad operativaSe ejecuta en front+gatewayRequiere ajustes acotadosDepende backend Boca
Control de promesaBoca controla end-to-endManejable front/gwFuera de control

Qué SÍ hacer

  • 3 protecciones core (cookie, stepper, queueit)
  • Mejoras anti-reventa y pago más rápido
  • Claridad sobre accesos y gestiones operativas
  • Visibilidad end-to-end (0% a 100%)
  • Reportería automática de fraude

Qué NO hacer

  • Features no vinculadas a seguridad
  • Tocar funciones que ya rinden bien
  • Modificar cupos, categorías o backend de Boca
  • Diseños que den lugar a memes
  • Bloquear socios automáticamente

SYSTEM LOOPS

Feedback loops

BocaOS no es lineal. Cada ciclo alimenta al siguiente.

Métricas → Estrategia

Cada release valida o invalida apuestas. Las métricas ajustan prioridades para el próximo trimestre.

Bug patterns → Backlog

Patrones recurrentes repriorizan automáticamente el backlog a través del triage técnico.

Cliente → Roadmap

El feedback durante la validación de software en testing ajusta el foco de las próximas iniciativas.

Eventos → Aprendizaje

Cada partido es un test real en producción. Los datos alimentan la estrategia operativa continua.
