FASE 1 · CONTEXTO 2026

El problema prioritario

Bypass técnico de QueueIT mediante cookie, URLs directas y acceso sin token válido. Los socios legítimos reportan no conseguir entradas sistemáticamente.

60-70%
ventas en primeros 5 min

Pico artificial, no refleja salida real de fila

<2 min
tiempo promedio compra

Velocidad que indica automatización (bots)

0%
visibilidad end-to-end

No existen métricas de transacciones hoy

Para quién resolvemos:
Prioridad 1: Socio activo sin filtro de asistencia que compite contra bots.
Prioridad 2: Socio con filtro que necesita mejor comunicación.

FASE 2 · ESTRATEGIA

Norte y 3 Apuestas Core

Objetivo: Cerrar los vectores de bypass de QueueIT y lograr que las ventas reflejen el orden real de acceso de socios legítimos.

1. Cookie Binding

Asociar cookie ↔ socio_id para eliminar 80% del fraude por cookies compartidas.

Target: <2% compartidas

Timeline: Q1 - Implementado

2. Stepper Obligatorio

Forzar pasos secuenciales para eliminar acceso directo a URLs de checkout.

Target: Ventas 5min <30%

Timeline: Q1 - Testear

3. Validación QueueIT

Validar token en TODOS los endpoints para cerrar el bypass de fila.

Target: 100% tokens bloqueados

Timeline: Q1 - Implementar

FASE 3 · REGLAS DE EJECUCIÓN

Límites explícitos

QUÉ SÍ HACER

  • Implementar las 3 protecciones core
  • Mejoras de seguridad contra reventa
  • Mejoras pago rápido / menos errores
  • Claridad sobre accesos o gestiones
  • Métricas end-to-end (visibilidad 100%)
  • Reportería automática de fraude

QUÉ NO HACER

  • Features ajenas a seguridad/anti-fraude
  • Tocar funciones que ya rinden bien
  • Modificar cupos, categorías o backend
  • Integración profunda con backend Boca
  • Cambios gráficos memeables
  • Bloquear socios automáticamente
