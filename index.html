KERNEL · APUESTAS

Las 3 apuestas core

¿Qué creemos que va a mover la aguja?

1. Cookie Binding

Si asociamos cookie ↔ socio_id, eliminaremos el 80% del fraude por cookie compartida.

  • › Binding en primer login
  • › Bloquear intentos re-uso

Target: cookies compartidas <2%

Timeline: Q1 — Ya implementado

2. Stepper Obligatorio

Si forzamos pasos secuenciales, eliminaremos acceso directo a urls como /checkout.

  • › State machine sever-side
  • › Token invalida al saltar pasos

Target: ventas 5min <30%

Timeline: Q1 — Implementar y testear

3. Validación QueueIT

Si validamos token en TODOS los endpoints críticos, cerramos bypass de fila.

  • › Middleware validación global
  • › Invalidar por inactividad

Target: Tokens inválidos detectados

Timeline: Q1 — Implementar

PRIORIZACIÓN ESTRICTA

Matriz Go / No-Go

Cómo decidimos qué entra al backlog para asegurar estabilidad y foco.

Criterio🟢 GO🟡 Condicional🔴 NO GO
Impacto en anti-fraudeReduce bypass o reventa directoImpacto indirecto o a validarSin impacto claro
Viabilidad operativaEcosistema actual (front+gw)Requiere ajustes acotadosDepende backend Boca
Control de promesaBoca controla end-to-endManejable front/gwFuera de control

Qué SÍ hacer

  • Las 3 protecciones core (cookie, stepper, queueit)
  • Mejoras de seguridad anti-reventa
  • Mejoras pago rápido/menos errores
  • Visibilidad end-to-end y reportes de fraude automáticos

Qué NO hacer

  • Nuevas features no vinculadas a seguridad
  • Tocar funciones que ya rinden bien
  • Modificar cupos, categorías o backend de Boca
  • Modificaciones gráficas memeables / bloqueos automáticos
