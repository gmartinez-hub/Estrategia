Cómo se planifica

Problema Prioritario

Bypass técnico de QueueIT mediante cookie, URLs directas y acceso sin token válido.

P1: Socio activo sin filtro (mayoría de la base)

P2: Socio con filtro que necesita mejor comunicación

KERNEL · APUESTAS

3 hipótesis core

Cookie Binding

Si asociamos cookie ↔ socio_id de forma permanente, eliminamos el 80% del fraude por cookie.

Validamos: cookies compartidas <2%

Timeline: Q1 — ya implementado

Stepper Obligatorio

Si forzamos recorrer pasos secuenciales con tokens de sesión, eliminamos acceso directo a /checkout.

Validamos: ventas 5min <30%

Timeline: Q1 — implementar

Validación QueueIT

Si validamos que el token de QueueIT sea válido en TODOS los endpoints, cerramos bypass de fila.

Validamos: correlación posición ↔ compra

Timeline: Q1 — implementar

KERNEL · FILTRO

Matriz Go / No-Go

Toda propuesta pasa por este filtro antes de entrar al runtime.

CriterioGOCondicionalNO GO
Impacto anti-fraudeReduce bypass directoImpacto indirectoSin impacto claro
Viabilidad operativaEcosistema actual (front+gw)Ajustes acotadosCambios estructurales
Control de promesaBoca controla end-to-endManejable con front y gwFuera de control

Qué SÍ hacer

  • 3 protecciones core (cookie, stepper, QueueIT)
  • Mejoras de seguridad contra reventa
  • Métricas end-to-end + reportería fraude
  • Mejoras de checkout (velocidad, menos errores)

Qué NO hacer

  • Features no relacionadas con seguridad
  • Cambios en backend de Boca
  • Bloquear socios automáticamente
  • Modificaciones gráficas sensibles

SCHEDULER · PROCESOS

Doble refinamiento

Cada funcionalidad nueva pasa por dos instancias de refinamiento.

Refi Técnica Backend

PM + Devs analizan dependencias, riesgos técnicos, complejidad y puntos de contacto con el ecosistema de Boca.

INPUT: Transcripción + Brief

OUTPUT: Resumen técnico en Notion

Refi UX/UI

PM + Design refinan la experiencia del socio. Escala con el tamaño de la iniciativa.

INPUT: Resumen técnico + contexto usuario

OUTPUT: Specs de diseño en Notion

RELEASE

Cómo se entrega

El release pipeline tiene 3 checkpoints antes de llegar al usuario final.

QA Interno

El equipo verifica criterios de aceptación antes de exponer al cliente.

Cliente Valida (SCOPE LOCK)

Software funcionando en testing. Scope se cierra. Ajustes menores, no features nuevas.

Deploy a Producción

Deploy + release notes en Jira + comunicación al cliente vía Task Reporting.

SYSTEM LOOPS

Feedback loops

BocaOS no es lineal. Cada ciclo alimenta al siguiente.

Métricas → Estrategia

Cada release valida o invalida apuestas. Las métricas ajustan prioridades.

Bug patterns → Backlog

Patrones recurrentes repriorización automáticamente el backlog.

Cliente → Roadmap

Feedback en testing ajusta foco de próximas iniciativas.

Eventos → Aprendizaje

Cada partido es un test en producción. Datos alimentan la estrategia.
